# 3일차 프로젝트 뜯어보기: 아키텍처 및 폴더 구조 분석

이 문서는 프로젝트의 전체적인 폴더 구조와 각 구성 요소의 역할을 정리한 팀 스터디용 자료입니다. FastAPI 기반의 레이어드 아키텍처(Layered Architecture) 및 현대적인 Python 개발 도구(`uv`)를 바탕으로 설계되었습니다.

---

## 1. 주요 디렉터리 역할 및 작성 내용

### 📂 app/core/
- **역할**: 애플리케이션의 핵심 설정 및 전역적으로 사용되는 공통 모듈을 관리합니다. 보안(JWT, 암호화), 전역 설정, 데이터베이스 연결 초기화 등이 포함됩니다.
- **주요 파일 및 작성 내용**:
  - `config.py`: 환경 변수(`.env`) 및 프로젝트 전역 설정 관리.
  - `db/databases.py`: SQLAlchemy 엔진 및 세션 팩토리(SessionLocal) 설정.
  - `db/models.py`: 데이터베이스 마이그레이션을 위한 기본 Base 모델 정의 및 엔티티 연결.

### 📂 app/models/
- **역할**: 데이터베이스 테이블과 1:1로 매핑되는 **SQLAlchemy ORM 모델**을 정의하는 계층입니다.
- **작성 내용**: 테이블 명, 컬럼 속성(PK, FK, 데이터 타입, 제약조건), 테이블 간의 관계(Relationship)를 작성합니다. 데이터베이스의 물리적 구조를 코드로 대변하는 영역입니다.

### 📂 app/repositories/
- **역할**: 데이터베이스에 직접 접근하여 CRUD(Create, Read, Update, Delete) 쿼리를 수행하는 **데이터 액세스 계층**입니다.
- **작성 내용**: SQLAlchemy 세션을 활용해 특정 모델의 데이터를 조회, 삽입, 수정, 삭제하는 저수준(Low-level) 쿼리 로직을 함수나 클래스로 구현합니다. 비즈니스 로직과 DB 접근 로직을 철저히 분리하기 위해 사용됩니다.

### 📂 app/schemas/
- **역할**: 데이터의 입력 및 출력 검증을 담당하는 **Pydantic 모델(DTO - Data Transfer Object)** 계층입니다.
- **작성 내용**: API 요청(Request Body) 검증용 스키마, API 응답(Response Body) 직렬화용 스키마를 작성합니다. DB 모델(models)과 분리하여 클라이언트에 보여줄 데이터 구조를 유연하게 제어합니다.

### 📂 app/services/
- **역할**: 핵심 **비즈니스 로직**을 처리하는 계층입니다.
- **작성 내용**: `repositories`에서 조회한 데이터를 가공하거나, 여러 리포지토리를 조합하여 하나의 비즈니스 흐름을 완성하는 고수준(High-level) 로직을 구현합니다. 외부 API 연동이나 비즈니스 규칙 검증, 트랜잭션 관리도 주로 여기서 처리합니다.

### 📂 app/apis/
- **역할**: 클라이언트의 요청을 받는 **엔드포인트(라우터)** 계층입니다.
- **작성 내용**: FastAPI의 `APIRouter`를 사용해 URL 경로(Route)를 정의하고, 클라이언트의 입력값(`schemas`)을 받아 적절한 `services`를 호출한 뒤 결과를 반환합니다. 이 계층은 단순 요청 접수 및 응답 반환 역할에 집중하며, 복잡한 로직은 포함하지 않는 것이 원칙입니다.

## 2. 주요 파일별 역할

### 📄 app/main.py
- **역할**: 애플리케이션의 **진입점(Entry Point)**입니다.
- **상세 내용**: FastAPI 인스턴스를 생성하고, CORS 미들웨어 설정, 전역 예외 처리(Exception Handler), 그리고 `app/apis/`에 정의된 라우터들을 등록(`include_router`)하는 작업을 수행합니다. UVicorn 등의 WAS가 이 파일을 로드하여 서버를 실행합니다.

### 📄 app/core/config.py
- **역할**: 시스템 **환경 설정 및 환경 변수 관리** 파일입니다.
- **상세 내용**: Pydantic의 `BaseSettings`를 활용하여 `.env` 파일의 환경 변수(DB 접속 정보, Secret Key, 외부 API 토큰 등)를 안전하게 읽어오고 타입 검증을 수행합니다. 시스템 전반에서 설정값을 참조할 때 이 파일을 거치게 됩니다.

### 📄 pyproject.toml
- **역할**: 프로젝트의 **메타데이터 및 의존성 설정 파일**입니다.
- **상세 내용**: Python 표준 프로젝트 설정 파일로, 프로젝트 이름, 버전, 필요한 라이브러리 목록(FastAPI, SQLAlchemy 등) 및 린터/포매터(Ruff 등)의 설정이 명시되어 있습니다. 현대 Python 생태계에서 `requirements.txt`와 `setup.py`를 대체합니다.

### 📄 uv.lock
- **역할**: 의존성의 **정확한 버전 및 설치 트리를 고정(Lock)**하는 파일입니다.
- **상세 내용**: 차세대 고속 Python 패키지 관리자인 `uv`에 의해 자동 생성됩니다. 프로젝트에 설치된 모든 패키지와 그 하위 의존성 라이브러리들의 정확한 버전, 체크섬 정보가 기록되어 있어, 팀원들 및 운영 환경(Docker 등)에서 항상 100% 동일한 개발 환경을 빌드할 수 있도록 보장합니다. *(직접 수정하지 않습니다.)*


## 3. 데이터베이스 연결 설정
FastAPI 애플리케이션과 데이터베이스(DB)를 안전하고 효율적으로 연결하기 위한 핵심 설정 파일입니다. 주로 **SQLAlchemy** 라이브러리를 사용하여 다음과 같은 핵심 요소들을 정의하고 관리합니다.

* **연결 통로 개설 (`create_engine`)**: 데이터베이스의 주소(URL)를 인자로 받아, 실제 DB와 애플리케이션 사이에 데이터를 주고받을 수 있는 실제 연결 통로(엔진)를 생성합니다.
* **대화 창구 관리 (`sessionmaker`)**: DB에 명령을 보내고 트랜잭션을 관리할 수 있는 개별적인 대화 창구인 **'세션(Session)'** 인스턴스를 찍어내는 공장 역할을 합니다.
* **기준 클래스 정의 (`DeclarativeBase`)**: 파이썬의 클래스를 데이터베이스의 테이블 구조와 매핑해 주는 ORM의 기준점이 되는 기반 클래스입니다. 모든 DB 모델 클래스는 이 Base를 상속받아 작성됩니다.
* **세션 주입 함수 (`get_db`)**: FastAPI의 `Depends`와 연동하여, API 요청이 들어올 때마다 세션을 안전하게 열고(`db = SessionLocal()`), 작업이 끝나면 자동으로 전화를 끊어 자원 누수를 방지하는(`db.close()`) 제너레이터 함수를 구현합니다.


## 4. SQLAlchemy 와 Alembic 흐름 
코드로 작성된 데이터베이스 모델을 실제 데이터베이스 테이블로 안전하게 반영하고 버전 제어를 하기 위해 **SQLAlchemy**와 **Alembic**을 함께 활용합니다. 전체적인 흐름은 다음과 같습니다.

1. **SQLAlchemy 모델 작성**
   * `app/models/` 디렉터리 내에 파이썬 클래스를 생성하고, 데이터베이스의 기준이 되는 `Base` 클래스를 상속받습니다.
   * `__tablename__`을 지정한 뒤, 테이블에 들어갈 컬럼명과 데이터 타입, 제약조건(기본키, Unique 등)을 코드로 정의합니다.

2. **Alembic 마이그레이션 파일 생성 (버전 생성)**
   * 파이썬 코드로 작성한 모델(테이블 구조)의 변경 사항을 감지하여 자동으로 이력을 기록하는 스크립트를 생성합니다.
   * 터미널에 아래 명령어를 입력하면 `alembic/versions/` 폴더 내에 변경 내역이 담긴 버전 파일이 만들어집니다.
     ```bash
     alembic revision --autogenerate -m "작성한 커밋 메시지"
     ```

3. **마이그레이션 실행 (DB에 테이블 생성)**
   * 생성된 버전 스크립트를 실행하여 실제 데이터베이스 서버에 테이블을 생성하거나 수정 사항을 최종 반영합니다.
   * 터미널에 아래 명령어를 실행하면 데이터베이스가 가장 최신 버전 상태(`head`)로 업데이트됩니다.
     ```bash
     alembic upgrade head
     ```

## 💡 API 구현 및 데이터 흐름 (Data Flow)

사용자가 API를 호출했을 때, 요청(Request)이 들어와서 데이터베이스를 거쳐 응답(Response)으로 나가는 전체적인 아키텍처 흐름은 다음과 같습니다.

### 📊 API 데이터 흐름 다이어그램

```text
  [ Client ] 
      │
      ▼  (1) Request (HTTP)
┌────────────────────────────────────────────────────────┐
│ 1. app/apis/ (Router)                                  │ ───► [ app/schemas/ ]
│    - 요청 진입점, URL 라우팅                           │      (Request 데이터 유효성 검사)
└─────────────────────────┬──────────────────────────────┘
                          │
                          ▼ (2) DTO / 파라미터 전달
┌────────────────────────────────────────────────────────┐
│ 2. app/services/ (Service Layer)                      │ ───► [ AI Model / External API ]
│    - 핵심 비즈니스 로직 처리, AI 모델 추론(Inference)   │
└─────────────────────────┬──────────────────────────────┘
                          │
                          ▼ (3) DB 작업 요청
┌────────────────────────────────────────────────────────┐
│ 3. app/repositories/ (Repository Layer)                │
│    - DB CRUD 쿼리 캡슐화, 데이터 접근 전담             │
└─────────────────────────┬──────────────────────────────┘
                          │
                          ▼ (4) ORM 매핑 및 쿼리 실행
┌────────────────────────────────────────────────────────┐
│ 4. app/models/ & app/core/db/ (Database Layer)         │
│    - SQLAlchemy ORM 객체 모델 및 실제 DB 세션          │
└────────────────────────────────────────────────────────┘
                          │
                          ▼ (5) 역순으로 데이터 반환 (Models ──► Repositories ──► Services)
┌────────────────────────────────────────────────────────┐
│ 5. app/apis/ (Response Serialization)                  │ ───► [ app/schemas/ ]
│    - 클라이언트에게 보낼 최종 응답 데이터 정제        │      (Response JSON 변환)
└─────────────────────────┬──────────────────────────────┘
                          │
                          ▼ (6) Response (JSON)
  [ Client ]

```

### 🔍 레이어별 역할 및 상세 흐름

#### **1. Client ➡️ API (요청 진입)**

* **역할:** 클라이언트(프론트엔드 등)가 특정 URL로 HTTP 요청을 보냅니다.
* **관련 디렉터리:** `app/apis/`, `app/schemas/`
* **상세 흐름:**
* `app/apis/`에 정의된 라우터(Router)가 요청을 받습니다.
* 이때 `app/schemas/`에 정의된 Pydantic 모델(Request Schema)을 통해 입력 데이터의 유효성 검사(Validation) 및 타입 체크가 자동으로 이루어집니다.



#### **2. API ➡️ Service (비즈니스 로직 수행)**

* **역할:** API 라우터는 들어온 데이터를 그대로 받아 실제 핵심 비즈니스 로직을 담당하는 서비스 레이어로 넘깁니다.
* **관련 디렉터리:** `app/services/`
* **상세 흐름:**
* 서비스 레이어는 데이터의 가공, 권한 체크, 외부 API 연동, **인공지능 모델 서빙(Inference)** 등 애플리케이션의 핵심 비즈니스 로직을 수행합니다.
* 컨트롤러(API)와 데이터 접근(Repository) 사이에서 중재자 역할을 하며, API가 데이터베이스 구조를 직접 알지 못하도록 결합도를 낮춥니다.



#### **3. Service ➡️ Repository (데이터베이스 접근)**

* **역할:** 비즈니스 로직을 처리하는 과정에서 DB 데이터가 필요하면, 데이터 엔티티에 직접 접근하는 레이어에 요청합니다.
* **관련 디렉터리:** `app/repositories/`
* **상세 흐름:**
* `app/repositories/`는 데이터베이스의 **CRUD(생성, 조회, 수정, 삭제) 쿼리문**을 캡슐화한 곳입니다.
* 서비스 레이어로부터 요청을 받아 **SQLAlchemy** 등을 이용해 DB에 명령을 내립니다.



#### **4. Repository ➡️ Database & Model (데이터 처리)**

* **역할:** 실제 DB와 맞닿아 데이터를 가져오거나 저장합니다.
* **관련 디렉터리:** `app/models/`, `app/core/db/`
* **상세 흐름:**
* `app/core/db/databases.py`에서 연결된 DB 세션을 활용합니다.
* `app/models/`에 정의된 SQLAlchemy ORM 객체(DB 엔티티)를 통해 데이터가 객체 형태로 매핑되어 레포지토리에 반환됩니다.



#### **5. Repository ➡️ Service ➡️ API ➡️ Client (응답 반환)**

* **역할:** 가공된 데이터를 역순으로 전달하여 최종적으로 사용자에게 응답합니다.
* **관련 디렉터리:** `app/schemas/`
* **상세 흐름:**
* DB에서 꺼내온 데이터(`models`)를 레포지토리가 서비스에 돌려줍니다.
* 서비스는 로직 처리를 완료한 후 API 라우터로 넘깁니다.
* API 라우터는 클라이언트에게 안전하고 정제된 데이터만 보여주기 위해 `app/schemas/`에 정의된 **Response Schema** 형태로 데이터를 변환(Serialization)하여 JSON 형식으로 최종 응답합니다.
