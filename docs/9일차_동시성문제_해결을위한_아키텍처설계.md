# FastAPI와 Redis를 이용한 Event-Driven Architecture 설계

> 동시성 문제 해결을 위한 아키텍처 설계 학습 정리

---

## 1. Event-Driven Architecture (EDA) 란?

### 개념
Event-Driven Architecture(이하 EDA)는 시스템 내의 컴포넌트들이 **이벤트(Event)** 를 통해 서로 통신하는 소프트웨어 아키텍처 패턴이다.

기존의 요청-응답(Request-Response) 방식과 달리, EDA에서는 특정 행위가 발생했을 때 이벤트를 발행(Publish)하고, 해당 이벤트를 구독(Subscribe)한 컴포넌트가 비동기적으로 처리한다.

### 핵심 구성 요소

| 구성 요소 | 설명 |
| --- | --- |
| **Event Producer** | 이벤트를 생성하고 발행하는 주체 (예: FastAPI 엔드포인트) |
| **Event Broker** | 이벤트를 수신하고 전달하는 중간 매개체 (예: Redis) |
| **Event Consumer** | 이벤트를 구독하고 처리하는 주체 (예: Background Worker) |

### 동작 흐름

```
[Client 요청]
     ↓
[FastAPI (Producer)]
     ↓ 이벤트 발행 (Publish)
[Redis (Broker)]
     ↓ 이벤트 전달 (Subscribe)
[Worker (Consumer)]
     ↓ 비동기 처리
[DB / 외부 서비스]
```

### EDA의 장점

- **느슨한 결합(Loose Coupling)**: Producer와 Consumer가 서로를 직접 알 필요 없음
- **확장성(Scalability)**: Consumer를 독립적으로 수평 확장 가능
- **비동기 처리**: 응답 지연 없이 빠른 응답 가능
- **장애 격리**: 한 컴포넌트의 장애가 전체 시스템에 영향을 주지 않음

### EDA의 단점

- 디버깅 및 추적이 어려움
- 이벤트 순서 보장이 어려울 수 있음
- 시스템 복잡도 증가

## 2. 동시성 문제란?

### 개념
동시성 문제(Concurrency Problem)는 여러 프로세스 또는 스레드가 **동시에 같은 자원에 접근**할 때 발생하는 문제다.

### 대표적인 동시성 문제

#### 1) Race Condition (경쟁 조건)
두 개 이상의 요청이 동시에 같은 데이터를 읽고 수정할 때 발생한다.

```
예시: 재고 차감
- 재고: 1개
- 사용자 A, B가 동시에 구매 요청
- A: 재고 1개 확인 → 구매 처리 → 재고 0으로 업데이트
- B: 재고 1개 확인 (A가 차감하기 전) → 구매 처리 → 재고 0으로 업데이트
- 결과: 재고가 -1이 되는 문제 발생
```

#### 2) Deadlock (교착 상태)
두 프로세스가 서로가 점유한 자원을 기다리며 무한 대기 상태에 빠지는 현상

#### 3) Data Inconsistency (데이터 불일치)
여러 서비스가 동시에 데이터를 수정하여 데이터가 일관성을 잃는 현상

## 3. Redis란?

### 개념
Redis(Remote Dictionary Server)는 **인메모리(In-Memory) 기반의 Key-Value 데이터 저장소**다. 디스크가 아닌 메모리에 데이터를 저장하기 때문에 매우 빠른 읽기/쓰기 성능을 제공한다.

### Redis의 주요 기능

| 기능 | 설명 | 활용 예시 |
| --- | --- | --- |
| **Cache** | 자주 사용하는 데이터를 메모리에 저장 | DB 조회 결과 캐싱 |
| **Pub/Sub** | 이벤트 발행/구독 메시징 시스템 | 실시간 알림, 채팅 |
| **Queue** | 작업 큐 관리 | 비동기 작업 처리 |
| **Distributed Lock** | 분산 환경에서의 잠금 처리 | 동시성 문제 해결 |
| **Session Store** | 사용자 세션 관리 | 로그인 상태 유지 |

### Redis의 데이터 구조

- **String**: 기본 Key-Value
- **List**: 순서가 있는 데이터 목록 (Queue 구현에 활용)
- **Set**: 중복 없는 집합
- **Sorted Set**: 점수 기반 정렬 집합
- **Hash**: 필드-값 쌍의 맵

## 4. Redis를 이용한 동시성 문제 해결

### 4-1. Distributed Lock (분산 락)

분산 환경에서 여러 인스턴스가 동시에 같은 자원에 접근하는 것을 방지한다.

#### Redis SETNX를 이용한 락 구현 원리
```
SETNX lock_key value  # 키가 없을 때만 SET (Set if Not eXists)
```
- 락 획득 성공: 해당 프로세스만 작업 진행
- 락 획득 실패: 다른 프로세스가 작업 중 → 대기 또는 재시도

#### Python (redis-py) 예시
```python
import redis
import time

r = redis.Redis(host='localhost', port=6379)

def acquire_lock(lock_name: str, timeout: int = 10) -> bool:
    """분산 락 획득"""
    result = r.set(lock_name, "locked", nx=True, ex=timeout)
    return result is not None

def release_lock(lock_name: str):
    """분산 락 해제"""
    r.delete(lock_name)

# 사용 예시
lock_name = "inventory:product_1"
if acquire_lock(lock_name):
    try:
        # 재고 차감 로직 수행
        pass
    finally:
        release_lock(lock_name)
else:
    # 락 획득 실패 처리
    raise Exception("다른 요청이 처리 중입니다.")
```

### 4-2. Redis Queue를 이용한 순차 처리

동시 요청을 큐에 넣어 순서대로 처리함으로써 Race Condition을 방지한다.

```
[동시 요청 A, B, C]
       ↓
[Redis Queue에 순서대로 적재]
       ↓
[Worker가 순차적으로 처리]
```

#### Python 예시
```python
import redis

r = redis.Redis(host='localhost', port=6379)

# 큐에 작업 추가 (Producer)
def enqueue_task(task_data: dict):
    r.rpush("task_queue", json.dumps(task_data))

# 큐에서 작업 처리 (Consumer)
def process_task():
    while True:
        task = r.blpop("task_queue", timeout=5)
        if task:
            data = json.loads(task[1])
            # 작업 처리
```

## 5. FastAPI + Redis를 이용한 EDA 구현

### 아키텍처 구성

```
[FastAPI App]
     ↓ 이벤트 발행
[Redis Pub/Sub or Queue]
     ↓ 이벤트 소비
[Background Worker]
     ↓
[MySQL DB]
```

### 5-1. Redis Pub/Sub 방식

#### Publisher (FastAPI)
```python
import redis.asyncio as aioredis
from fastapi import FastAPI
import json

app = FastAPI()
redis = aioredis.from_url("redis://localhost:6379")

@app.post("/api/v1/diagnosis/")
async def create_diagnosis(data: dict):
    # 이벤트 발행
    event = {
        "event_type": "diagnosis_requested",
        "payload": data
    }
    await redis.publish("diagnosis_channel", json.dumps(event))
    return {"message": "진단 요청이 접수되었습니다."}
```

#### Subscriber (Worker)
```python
import redis.asyncio as aioredis
import asyncio
import json

async def subscribe():
    redis = aioredis.from_url("redis://localhost:6379")
    pubsub = redis.pubsub()
    await pubsub.subscribe("diagnosis_channel")

    async for message in pubsub.listen():
        if message["type"] == "message":
            event = json.loads(message["data"])
            await handle_event(event)

async def handle_event(event: dict):
    if event["event_type"] == "diagnosis_requested":
        # AI 진단 처리 로직
        pass

asyncio.run(subscribe())
```

### 5-2. Redis Queue (작업 큐) 방식

#### Producer (FastAPI)
```python
from fastapi import FastAPI
import redis.asyncio as aioredis
import json

app = FastAPI()
redis = aioredis.from_url("redis://localhost:6379")

@app.post("/api/v1/xray/analyze/")
async def analyze_xray(file_id: int):
    task = {
        "task_type": "xray_analysis",
        "file_id": file_id
    }
    # 작업 큐에 추가
    await redis.rpush("xray_queue", json.dumps(task))
    return {"message": "분석 요청이 큐에 추가되었습니다.", "file_id": file_id}
```

#### Consumer (Worker)
```python
import redis.asyncio as aioredis
import asyncio
import json

async def worker():
    redis = aioredis.from_url("redis://localhost:6379")

    while True:
        # 블로킹 방식으로 큐에서 작업 꺼내기
        task_data = await redis.blpop("xray_queue", timeout=5)
        if task_data:
            task = json.loads(task_data[1])
            await process_xray(task)

async def process_xray(task: dict):
    file_id = task["file_id"]
    # AI 모델 분석 처리
    print(f"X-Ray 분석 중: file_id={file_id}")

asyncio.run(worker())
```

## 6. 흉부 X-Ray AI 진단 서비스에서의 EDA 적용

### 적용 시나리오

흉부 X-Ray AI 진단 서비스에서 여러 사용자가 동시에 진단을 요청하는 경우, AI 모델 추론은 GPU 자원을 많이 사용하므로 동시 처리 시 성능 저하가 발생할 수 있다. EDA를 적용하면 이를 효율적으로 처리할 수 있다.

### 아키텍처 다이어그램

```
[의료진 / 연구진]
      ↓ X-Ray 이미지 업로드 및 진단 요청
[FastAPI (Producer)]
      ↓ 진단 요청 이벤트 발행
[Redis Queue]
      ↓ 순차적으로 작업 전달
[AI Worker (Consumer)]
      ↓ AI 모델 추론
[결과 저장 (MySQL)]
      ↓ 완료 이벤트 발행
[Redis Pub/Sub]
      ↓ 결과 알림
[FastAPI → Client]
```

### 동시성 문제 해결 포인트

| 문제 | 해결 방법 |
| --- | --- |
| 동시 진단 요청으로 인한 GPU 과부하 | Redis Queue로 순차 처리 |
| 같은 이미지 중복 분석 | Redis Lock으로 중복 방지 |
| 결과 조회 응답 지연 | Redis Cache로 결과 캐싱 |

## 7. 정리

| 항목 | 내용 |
| --- | --- |
| **EDA** | 이벤트 기반 비동기 아키텍처로 느슨한 결합과 확장성 제공 |
| **Redis 역할** | Event Broker, Queue, Distributed Lock, Cache |
| **동시성 해결** | Distributed Lock으로 Race Condition 방지, Queue로 순차 처리 |
| **FastAPI 연동** | `redis.asyncio`를 활용한 비동기 Pub/Sub 및 Queue 구현 |

EDA는 흉부 X-Ray AI 진단 서비스처럼 **무거운 AI 추론 작업을 비동기로 처리**해야 하는 시스템에 적합한 아키텍처 패턴이다. Redis를 Event Broker로 활용하면 동시성 문제를 효과적으로 해결하면서도 빠른 응답성을 유지할 수 있다.
