## 🏗️ 1. 브랜치 운영 구조

우리 팀은 빠른 개발과 유연한 배포가 필요한 단기 프로젝트의 특성에 맞추어 **GitHub Flow** 전략을 기반으로 운영합니다.

* **`main` 브랜치:** 언제든 제품으로 출시(배포) 가능한 상태를 유지하는 중심 브랜치입니다.
* **`feature/기능명` 브랜치:** 각자 맡은 기능을 개발하는 브랜치입니다.
* *예시:* `feature/login`, `feature/board-ui`, `feature/payment`

## 🔄 2. 작업 프로세스 (Work Flow)

1. **최신 코드 동기화 및 브랜치 생성**
새로운 작업을 시작할 때는 항상 `main` 브랜치에서 최신 코드를 당겨온 후, 자신의 기능 브랜치를 생성합니다.
```bash
git checkout main
git pull origin main
git checkout -b feature/기능명

```


2. **1일 1커밋 및 원격 푸시**
게을러지지 않기 위해 **하루 최소 1커밋** 규칙을 지키며 자신의 원격 레포지토리에 푸시합니다. 커밋을 진행할 때 팀 내 컨벤션 유지를 위해 하단의 커밋 템플릿을 활용합니다.
```bash
git add .
git commit -m "✨ feat: 로그인 입력 유효성 검사 추가"
git push origin feature/기능명

```


3. **Pull Request (PR) 생성 및 코드 설명**
기능 구현이 완료되면 `main` 브랜치를 향해 Pull Request(PR)를 생성합니다. PR을 작성할 때는 팀원들이 코드를 쉽게 이해할 수 있도록 아래 내용을 포함하여 작성합니다.
> 💡 **PR 작성 가이드 (코드 설명)**
> * **작업 내용:** 이번 PR에서 구현한 핵심 기능 요약
> * **주요 변경 사항:** 수정되거나 새롭게 추가된 파일 및 로직 설명
> * **리뷰어에게 바라는 점:** 특히 신경 써서 봐주었으면 하는 부분이나 고민되었던 지점
> 
> 


4. **코드 리뷰 및 병합 (Merge)**
팀원들에게 코드 리뷰를 요청하고, 최소 1명 이상의 승인(Approve)을 받은 후 `main` 브랜치에 Merge(병합)합니다.
5. **작업 브랜치 삭제**
병합이 완료된 `feature` 브랜치는 Git 레포지토리를 깔끔하게 관리하기 위해 즉시 삭제합니다.

## 💬 3. 커밋 메시지 컨벤션 (Commit Message Convention)

코드 히스토리를 직관적으로 알아보기 위해 커밋 메시지 첫머리에 아래 말머리(이모지 포함)를 필수 기재합니다.

| 이모지 & 태그 | 설명 |
| --- | --- |
| ✨ `feat` | 새로운 기능 추가 |
| 🐛 `fix` | 버그 수정 |
| 💡 `chore` | 기능 추가 없이 코드 수정 (오타, 주석, 패키지 매니저 설정 등) |
| 📝 `docs` | 문서 수정 (README, 팀 규칙 문서 등) |
| 🚚 `build` | 빌드 관련 파일 파일 수정 |
| ✅ `test` | 테스트 코드 추가/변경 (프로덕션 코드 변경 없음) |
| ♻️ `refactor` | 리팩터링 (기능 변화 없이 코드 구조만 개선) |
| 🚑 `hotfix` | 프로덕션(배포 환경) 긴급 수정 |

### ✍️ 커밋 메시지 작성 예시

* `✨ feat: 회원가입 이메일 인증 기능 구현`
* `📝 docs: 2일차 git branch 전략 문서 추가`
* `🐛 fix: 메인 페이지 컴포넌트 렌더링 에러 수정`
